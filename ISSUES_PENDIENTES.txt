15. [PENDIENTE] Bypass de validacion de certificados TLS
    -----------------------------------------------------
    Archivo: Program.cs, Linea 92
    Severidad: CRITICA

    Problema:
    ```csharp
    ServerCertificateCustomValidationCallback =
        HttpClientHandler.DangerousAcceptAnyServerCertificateValidator
    ```

    La aplicacion deshabilita TODA validacion de certificados SSL/TLS para
    conexiones AFIP. Esto hace vulnerable la aplicacion a ataques MITM.

    Solucion:
    - Remover DangerousAcceptAnyServerCertificateValidator
    - Configurar validacion de certificados correctamente para AFIP
    - Solo usar en desarrollo con advertencias explicitas

33. [PENDIENTE] Concepto fijo en "Productos" - No soporta Servicios
    ----------------------------------------------------------------
    Archivo: Services/ARCAApiService.cs, Linea 268
    Severidad: BAJA (solo si se venden servicios)

    Problema:
    El sistema siempre envia `Concepto = "Productos"` (codigo 1) a AFIP.
    No soporta facturacion de servicios ni productos+servicios.

    AFIP permite 3 valores de Concepto:
    | Concepto             | Codigo | Campos Adicionales Requeridos        |
    |----------------------|--------|--------------------------------------|
    | Productos            | 1      | Ninguno                              |
    | Servicios            | 2      | FchServDesde, FchServHasta, FchVtoPago |
    | Productos y Servicios| 3      | FchServDesde, FchServHasta, FchVtoPago |

    Para servicios, AFIP exige:
    - FchServDesde: Fecha inicio del periodo de servicio (formato yyyyMMdd)
    - FchServHasta: Fecha fin del periodo de servicio (formato yyyyMMdd)
    - FchVtoPago: Fecha de vencimiento del pago (formato yyyyMMdd)

    Solucion propuesta:
    1. Agregar campo VenConcepto (int?) a modelo Venta
    2. Agregar campos VenFchServDesde, VenFchServHasta, VenFchVtoPago (DateTime?)
    3. Modificar vista Facturas/Generar para permitir seleccionar concepto
    4. Si concepto = 2 o 3, mostrar campos de fecha de servicio
    5. Modificar ARCAApiService para enviar concepto y fechas dinamicamente
    6. Agregar validacion: si concepto != 1, fechas son obligatorias

    Impacto: Solo afecta si se venden servicios (consultorias, alquileres,
    mantenimiento, etc.). Para venta de productos fisicos, no se requiere cambio.

34. [PENDIENTE] Tributos siempre en cero - No soporta IIBB ni otros
    ----------------------------------------------------------------
    Archivo: Services/ARCAApiService.cs, Linea 266
    Severidad: BAJA (solo si se requiere discriminar tributos)

    Problema:
    El sistema siempre envia `ImporteTributos = 0` a AFIP.
    No permite discriminar tributos adicionales al IVA en las facturas.

    Tributos soportados por AFIP:
    | Codigo | Tributo                | Ejemplo                      |
    |--------|------------------------|------------------------------|
    | 1      | Impuestos Nacionales   | Impuesto PAIS                |
    | 2      | Impuestos Provinciales | IIBB (Ingresos Brutos)       |
    | 3      | Impuestos Municipales  | Tasa seguridad e higiene     |
    | 4      | Impuestos Internos     | Cigarrillos, bebidas         |
    | 99     | Otros                  | Percepciones, retenciones    |

    Estructura requerida por AFIP para tributos:
    ```json
    {
      "Tributos": [
        {
          "Id": 2,           // Tipo de tributo (codigo AFIP)
          "Desc": "IIBB",    // Descripcion del tributo
          "BaseImp": 1000,   // Base imponible
          "Alic": 3.5,       // Alicuota (porcentaje)
          "Importe": 35      // Importe calculado
        }
      ]
    }
    ```

    Solucion propuesta:
    1. Crear tabla TRIBUTOS con Id, Descripcion, Alicuota, Activo
    2. Crear tabla ARTICULO_TRIBUTOS (relacion N:N con articulos)
    3. O agregar campo TributoCod a Articulo si es uno solo por articulo
    4. Modificar ARCAApiService para calcular y enviar tributos
    5. Modificar vista de factura para mostrar tributos discriminados
    6. Agregar configuracion de agente de retencion/percepcion

    Impacto: Solo afecta si la empresa es agente de retencion/percepcion
    de IIBB o necesita discriminar otros tributos. En comercio minorista
    tipico, generalmente no se requiere.

    Nota: Si solo se necesita agregar percepcion de IIBB, se puede
    simplificar agregando un campo de alicuota IIBB al cliente y
    calculandolo automaticamente sobre el neto.
